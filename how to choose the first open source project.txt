何如选择并贡献于您的第一个开源项目
by  Katrina Owen

您的第一个贡献可能是艰巨的。它很像走进一个满是陌生人的房间，他们大部分人似乎相互认识。对于贡献这件事，你不知道他们内部的笑话或者外部的。他们涉及你从来没有听过的流行文化。你不知道，当走进一群人的时候，是否有人会轻微走到一边让你进入他们的谈话。可能他们会给你一个糟糕的印象。他们将告诉你你穿了一件糟糕的夹克。你的眼睛是错误的颜色。你发的gif是错误的。

这不仅仅是你的第一个pull请求.即使打开一个问题也是令人恐慌的。你遇到的问题是一个真正的问题吗？可能他们会告诉你，你是愚蠢的，因为你被什么小玩意蒙蔽了。

作为Exercism的维护者，我看到数以千计的人开始用新的编程语言，提供并收到关于代码的反馈，并且开始贡献于开源项目。你的第一个贡献可能是胆小的，但有正确的指导，我确信你将乐于成为开源社区的一份子，像我一样。

这里是我挑选出来的一些资源，帮助你选择正确的第一个项目，并且在提高你技能的同时做出有影响力的贡献。

Start with a project you use
开始贡献的最好的方法是从您已经使用的开源项目开始。
从您依赖的项目开始，有如下一些好处：
1、您更熟悉这个工程做什么的，它怎么工作
2、您对项目的提高和成果投入更多
3、您同项目有一个持续关系，以至您更可能不断回来

想想您喜欢的库，模块、插件或工具。查看您的工作项目或副项目的依赖列表。找到存储库并浏览源码，文档和开放的问题去获得项目可能是什么的一个印象。

您不必追求您能想到的最大的，最给人印象深刻的开源项目。一个小的项目有时候能更容易的盘绕在您的头脑中，并且维护者可能欣赏您的贡献，总的来说，更多因为他们得到较少的贡献。

没有方法提前知道一个项目是否将接受您的贡献，但您做两件事能增加您的机会，有一个良好的第一经验。

第一件事是寻找项目欢迎贡献的签名，第二件事是弄清他们在贡献中寻找什么。

Evaluate the project
通过观察两件事：活跃度和交流风格，您能获得一个十分良好的直觉，判断是否您想参与其中。

看提交列表，有最近的提交吗？它们有多频繁？提交大多是来自一个人还是有数个人参与？

现在看看一些问题。用一个开放的问题开始。有多少个问题？有新的吗？有非常久的吗？它们有维护者响应吗？有充满活力的讨论吗？点进已经关闭的问题。它们有最近关闭的吗？

接下来对pull请求做相同的操作。

这不是一门精确的学科，并且你找不到任何具体的数字。即使没有很多活动，它可能任然是一个很好的去贡献的项目。它可能是一个成熟、稳定的项目，不需要很多改动。但如果有很多问题并且pull请求已经被打开时了很长时间，且维护者都没响应，可以对它亮红灯了。

接下来，看看开放和关闭问题的谈论和pull请求。用最多评论排序。了解人们怎么交流。谈论是友好的吗？他们有礼貌的处理分歧吗？维护者对贡献表示感激吗？

有些维护者通过友好的信息或表情，确保新的贡献者感到受欢迎和欣赏。

花些时间阅读旧的问题或者邮件列表文档，如果他们有IRC的话，潜伏在里面，并且收集关于项目怎么工作的信息。对项目的社区有一个良好的感觉并决定是否你将愿意花费时间同其他参与者。

Dip your toes in
一旦你发现了一个你想贡献的项目，查阅它的README.README可能提及项目的贡献指南---经常被命名为CONTRIBUTING.md。如果README里没有提及，你可以看看文件目录。

没有标准规定人们放什么在这些文件里，然而是一个好的机会告诉其他人，他们正在寻找什么样的贡献。
另一个项目寻找帮助的地方就是去点开“Issues”标签，然后寻找标有“help wanted”或“good first issue”的问题。

Don't feel rushed to open an issue or pull request

当你觉得准备好做出贡献的时候，最好开始一个小的贡献。较小的贡献帮助项目的维护者更好的了解你，并且他们更可能去接受一个主要特性的变化或者开发。

对于开始贡献来说，最好的方式之一是帮助分类问题。当新bug报告进来的时候，看看你能否复现那个状态，并评论你的发现。

如果你一直在潜伏，你将开始关注重复的问题，以及固定复现但从未解决的问题。人们会问你知道答案的问题。你们把他们指向文档。或者如果没有文档，你能向文档中提交一个建议的修改。

文档可能是一些最容易和最难去做的贡献。有的是低级的拼写或语法错误。另一个经常发生的文档更改是重新构建一个部分，以提高清晰度和流程，或者更新文档以添加您遇到的一个点。更具挑战的文档修改经常涉及重构或者编写指南。

用分类和修改文档开始的好处是你能做出贡献。

Learn how others did it
如果你一直想去做出你的第一个贡献，但觉得紧张，你需要知道你不孤单。即使专业的软件开发者也能感到开源的可怕，因为它涉及跟从未谋面的人进行合作。

阅读别人的经验以减少的这种恐惧因素。从专业的软件开发者到那些刚刚开始的，每个人都有他们自己第一次贡献的故事。
